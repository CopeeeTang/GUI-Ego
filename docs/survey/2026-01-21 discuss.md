# 智能眼镜生成式 UI 研究项目实施文档

---

## 1. 背景介绍

### 1.1 研究领域概述

**Generative UI (GenUI)** 是指由 AI 根据用户意图和上下文动态生成用户界面的技术范式。与传统固定模板 UI 不同，GenUI 能够：
- 根据用户当前任务自适应调整界面元素
- 将复杂信息转化为可交互的视觉组件
- 减少用户的认知负担和操作步骤
### 1.2 现有技术进展

**Web/移动端 GenUI 已趋成熟**：

| 技术方向           | 代表工作                          | 成熟度   |
| -------------- | ----------------------------- | ----- |
| Design-to-Code | ScreenCoder, Design2Code      | ★★★★☆ |
| LLM 生成 UI      | Google GenUI (Gemini), v0.dev | ★★★★☆ |
| Agent 生成界面     | A2UI Protocol (Google)        | ★★★☆☆ |

**评估体系**：
- **PAGEN** (Google): 专家制作的 UI 生成评估数据集
- **Design2Code Benchmark**: 设计稿到代码的端到端评估
- 用户偏好研究：44% 情况下 LLM 生成 UI 与人类专家相当

### 1.3 现象级 Gap：智能眼镜场景的空白

**核心观察**：当前 GenUI 研究完全聚焦于屏幕场景（手机/电脑/网页），智能眼镜场景处于空白状态。

**Gap 的本质差异**：

| 维度        | 手机/电脑       | 智能眼镜                      |
| --------- | ----------- | ------------------------- |
| **用户注意力** | 集中在屏幕       | 分散在现实世界                   |
| **上下文来源** | 屏幕内容 + 用户输入 | 视觉场景 + 注视点 + 活动状态 + 语音    |
| **交互模式**  | 触摸/键鼠       | 注视 + 语音 + 手势              |
| **显示约束**  | 无严格限制       | 需 2 秒内可理解 (Glanceability) |
| **时机敏感性** | 用户主动触发      | 系统需判断何时介入                 |

**关键洞察**：
> "手机/电脑的 GUI 已经很完善，人的注意力全在屏幕上；只有当注意力无法集中时（智能眼镜）才需要 Generative UI"

## 2. 问题定义

### 2.1 核心研究问题（生成什么样的UI）

Pipeline：原始数据（context：ego，眼动，手势...）-> VLM理解(MODEL1：streaming video understanding) -> GUI生成 （MODEL2： GUI Code Generation） -> 部署到某一款智能眼镜上（先采一批数据）

现在的WebUI pipeline ：context -> GUI （Google Gemini + Nano banana）

基于pipeline，在Smart Glasses场景下可存在以下GAP，并可能有以下研究问题/工作进行：

#### RQ1: Context-to-UI Mapping（上下文到 UI 的映射）

**GAP1：模态扩展，上下文变多**
> 给定多模态上下文（场景视觉 + 注视点 + 活动状态），如何生成符合智能眼镜显示约束的 UI 组件？

问题映射模块：Streaming Video Understanding


**子问题拆解**：
- **RQ1.1 What to Show**: 根据上下文决定显示哪些信息元素
- **RQ1.2 How to Format**: 选择合适的 UI 原语（标签/卡片/指示箭头）


#### RQ2: Attention-Aware UI Complexity（注意力感知的 UI 复杂度）

**GAP2:注意力分散，眼动数据映射注意力**
> 如何根据用户当前的注意力状态（眼动）动态调整 UI 的复杂度和介入时机？

**子问题拆解**：
- **RQ2.1 When to Show**: 判断介入时机（用户空闲 vs 专注任务）
- **RQ2.2 Complexity Level**: 注意力低时简化 UI，注意力高时可展示详情
- **RQ2.3 Duration & Fade**: 显示持续时间和消失策略

#### RQ3: Spatial GEN UI in Real World（真实场景下的3D GenUI）

- **RQ3.1 Where to Show**: 空间锚定（世界坐标 vs 屏幕坐标 vs 物体绑定）
### 2.3 问题边界

**本研究聚焦**：
- ✅ 轻量级 AR 眼镜（带显示屏）
- ✅ 辅助现实场景（购物、导航、学习）
- ✅ 2D UI 组件生成

**讨论**：
主要为硬件层面的工程问题：
1.Smart Glasses下显示屏的问题
2.链路能否打通，是否有愿意合作的眼镜厂商
3.Smart Glasses是否有虚拟的类似VLA中simulator的模拟器
4.生成数据的评估问题（人工
5.生成UI渲染协议的问题？

### 3.2 核心技术模块

#### 3.2.1 多模态上下文编码器

**输入**：
- 场景图像（第一人称视角）
- 注视点坐标（Eye Tracking）
- 活动状态标签（walking, shopping, cooking...）
- 可选：语音查询

**输出**：
- 场景理解向量
- 注意焦点区域
- 用户意图推断

**技术选型**：
- 基座模型：LLaVA / Qwen-VL / InternVL
- 注视点融合：Gaze-conditioned attention

#### 3.2.2 UI Schema 选择器

**A2UI 扩展 Schema**（基于 Google A2UI 协议）：

```json
{
  "type": "surfaceUpdate",
  "surfaceId": "smart-glass-overlay",
  "components": [
    {
      "type": "ar.label",
      "properties": {
        "text": "新鲜度: 8.5/10",
        "spatial": {
          "anchorType": "object",
          "objectId": "orange-box-1",
          "billboarding": true
        },
        "attention": {
          "trigger": "gaze",
          "gazeDuration": 1.5,
          "priority": "medium",
          "autoHide": 5.0
        }
      }
    }
  ]
}
```

**支持的 UI 原语**：

| 原语类型 | 适用场景 | 复杂度 |
|---------|---------|--------|
| `ar.label` | 物体识别、价格标注 | Low |
| `ar.arrow` | 导航、物品定位 | Low |
| `ar.minimalTimer` | 烹饪、专注模式 | Low |
| `ar.comparisonCard` | 商品比较 | Medium |
| `ar.progressRing` | 长期任务追踪 | Medium |
| `ar.contextMenu` | 多选项决策 | High |

#### 3.2.3 注意力感知模块

**注意力状态分类**：

| 状态 | 特征 | UI 策略 |
|------|------|---------|
| **Idle** | 注视点稳定、无明确任务 | 可主动推荐 |
| **Browsing** | 注视点游移、探索性 | 渐进式展示 |
| **Focused** | 注视点集中、任务进行中 | 最小化打扰 |
| **Conversing** | 与人对话中 | 暂停所有 UI |

**触发机制**：
- Gaze Trigger: 注视物体 >1.5s 触发标签
- Voice Trigger: 语音命令立即响应
- Auto Trigger: 仅在 Idle 状态启用

### 3.3 渐进式 UI 展示策略

**以超市购物场景为例**：

```
阶段1: 基础标签 (用户注视橘子 >1.5s)
    → 显示 "🍊 新鲜度 9.2" 浮动标签

阶段2: 展开详情 (继续注视 >3.5s)
    → 展开营养信息卡片

阶段3: 对比视图 (语音 "比较")
    → 显示三盒橘子并排对比表

阶段4: 确认选择 (语音 "选这个")
    → "✓ 已加入购物篮" 提示
    → 所有 UI 自动隐藏
```

## 4. 评估指标

### 4.1 现有评估方法的局限

**Web GenUI 评估方法**（不可直接迁移）：
- Visual Similarity Score（视觉相似度）
- Code Accuracy（代码正确性）
- User Preference（用户偏好）

**为什么不适用于智能眼镜**：
- 缺乏 Glanceability 评估
- 缺乏时机判断准确性
- 缺乏注意力负载评估

### 4.2 提出的评估框架

#### 4.2.1 自动评估指标

| 指标 | 定义 | 计算方法 |
|------|------|---------|
| **Context-UI Relevance** | 生成 UI 与上下文的相关性 | VLM 评分 / 人工标注一致性 |
| **Information Compression** | 信息压缩效率 | 原始信息量 / UI 文本长度 |
| **Glanceability Score** | 2 秒内可理解程度 | 用户实验 + 眼动指标 |
| **Trigger Precision** | 触发时机的准确性 | 正确触发数 / 总触发数 |
| **Attention Load** | UI 对注意力的占用 | 注视点在 UI 上的停留时间 |

#### 4.2.2 用户实验指标

| 指标 | 测量方法 |
|------|---------|
| **Task Completion Rate** | 使用 UI 完成任务的成功率 |
| **Decision Time** | 从 UI 显示到用户决策的时间 |
| **User Preference** | A/B 测试（GenUI vs 文本推荐） |
| **Interruption Cost** | 主观打扰程度评分 (1-5) |

#### 4.2.3 Benchmark 构建计划

**数据来源**：P1_YuePan 数据集
- 3.17 小时眼动数据
- 330+ 条推荐记录
- 12% 用户接受率（基线）

**标注任务**：
1. 对每条推荐标注"理想 UI 形式"
2. 标注"最佳触发时机"
3. 标注"UI 复杂度等级"

**评估集划分**：
- Train: 60% (用于模型微调)
- Val: 20% (超参调优)
- Test: 20% (最终评估)

## 6. 风险与应对

| 风险 | 可能性 | 影响 | 应对策略 |
|------|--------|------|---------|
| 数据集规模不足 | 中 | 高 | 使用合成数据增强 |
| UI 渲染器适配问题 | 低 | 中 | 优先使用模拟器评估 |
| 用户实验招募困难 | 中 | 中 | 线上 Wizard-of-Oz 实验 |
| 与已有工作差异不明显 | 低 | 高 | 强调 attention-aware 的创新点 |
